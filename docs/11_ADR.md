# 11. Architecture Decision Records

Этот файл фиксирует ключевые архитектурные решения проекта.

Формат:

* статус;
* контекст;
* решение;
* последствия.

Новые решения добавлять в конец файла. Не переписывать историю задним числом.

## ADR-001: Используем SQLite для MVP

Статус: принято

Контекст:

Проект находится в MVP-стадии. Нужен простой, надёжный и быстрый способ хранить данные без отдельного сервера БД.

Решение:

Использовать SQLite через `better-sqlite3`.

Последствия:

Плюсы:

* простой локальный запуск;
* простой деплой;
* один файл БД;
* не нужен отдельный PostgreSQL-сервер;
* достаточно для первых пользователей.

Минусы:

* нужно аккуратно делать backup;
* нужно следить за доступом к файлу;
* при росте нагрузки может понадобиться PostgreSQL.

## ADR-002: Не используем ORM

Статус: принято

Контекст:

Проект небольшой, схема контролируемая, SQL понятный. ORM добавит лишний слой и усложнит отладку миграций.

Решение:

Использовать чистый SQL и repositories поверх `better-sqlite3`.

Последствия:

Плюсы:

* явные запросы;
* простая схема;
* меньше магии;
* проще понимать миграции.

Минусы:

* больше ручного mapping-кода;
* нужно внимательнее следить за типами и nullable-полями.

## ADR-003: Админка server-rendered HTML на Fastify

Статус: принято

Контекст:

Админка нужна для Олега и администратора проекта. На MVP не нужен сложный frontend.

Решение:

Делать админку на Fastify с server-rendered HTML.

Не использовать React/Vue.

Последствия:

Плюсы:

* нет отдельной сборки фронтенда;
* меньше зависимостей;
* быстрее разработка;
* проще деплой.

Минусы:

* UI будет проще;
* сложные интерактивные сценарии делать менее удобно.

## ADR-004: Один Node.js-процесс для бота и админки на MVP

Статус: принято

Контекст:

Бот и админка используют одну SQLite-базу. MVP должен быть простым в деплое и сопровождении.

Решение:

Запускать Telegram-бот и Fastify-админку в одном Node.js-процессе.

Последствия:

Плюсы:

* простой запуск;
* простой деплой;
* одна точка конфигурации;
* один доступ к SQLite.

Минусы:

* падение процесса затронет и бота, и админку;
* позже может понадобиться разделение процессов.

## ADR-005: Авторизация менеджеров через одноразовые коды

Статус: принято

Контекст:

Менеджеры должны привязываться к салонам без ручного ввода телефона и без сложной регистрации.

Решение:

Использовать одноразовые коды доступа из `manager_auth_codes`.

Код связан с `manager_id`. После ввода кода бот создаёт запись в `telegram_users` и помечает код использованным.

Последствия:

Плюсы:

* простой onboarding менеджера;
* не нужно спрашивать салон вручную;
* Telegram user id становится стабильной привязкой.

Минусы:

* Олегу или администратору нужно заранее создать код;
* нужен механизм сброса кода;
* коды нужно защищать от утечки.

## ADR-006: Snapshot-поля в заявках обязательны

Статус: принято

Контекст:

Справочники салонов и менеджеров будут меняться. Старые заявки должны оставаться такими, какими были на момент создания.

Решение:

Сохранять snapshot-поля в `orders` и `order_items`.

Примеры:

* `salon_name_snapshot`;
* `salon_email_snapshot`;
* `manager_name_snapshot`;
* `manager_phone_snapshot`;
* `manager_role_snapshot`;
* `item_name_snapshot`;
* `unit_price_snapshot`;
* `price_text_snapshot`;
* `card_text_snapshot`.

Последствия:

Плюсы:

* исторические заявки стабильны;
* карточки не ломаются при изменении справочников;
* проще разбирать старые спорные случаи.

Минусы:

* данные частично дублируются;
* нужно следить, чтобы при создании заявки snapshot заполнялся корректно.

## ADR-007: `orders.final_price` пока не используется

Статус: принято

Контекст:

Итоговая стоимость замера зависит от фактической ситуации на объекте. Бот на этапе заявки не знает всех деталей.

Решение:

Оставить `orders.final_price` в схеме, но не рассчитывать и не заполнять его.

Карточка должна показывать тарифы, но не итог.

Последствия:

Плюсы:

* не создаём ложную точность;
* не спорим с фактическим расчётом замерщика;
* оставляем поле для будущего.

Минусы:

* в админке нужно аккуратно называть ценовые колонки;
* позже потребуется отдельное решение, если захотим считать estimate.

## ADR-008: Правила салонов хранятся в `salon_required_items`

Статус: принято

Контекст:

У некоторых салонов есть обязательные позиции, которые должны попадать в каждую заявку. Это не комментарий менеджера, а правило салона.

Решение:

Хранить обязательные позиции в `salon_required_items`.

При создании заявки активные записи с `auto_add_to_order = 1` копируются в `order_items` со snapshot-полями.

Последствия:

Плюсы:

* правила салона формализованы;
* бот применяет их автоматически;
* карточки стабильны;
* старые заявки сохраняют старый текст правила.

Минусы:

* нужно поддерживать UI управления правилами;
* нужно аккуратно различать пользовательские и автоматические позиции.

## ADR-009: Фиксированная оплата салона через `default_payment_by` и `is_payment_by_fixed`

Статус: принято

Контекст:

У некоторых салонов оплата всегда фиксирована. Менеджера не нужно каждый раз спрашивать, кто оплачивает замер.

Решение:

Использовать поля:

* `default_payment_by`;
* `is_payment_by_fixed`.

Если `is_payment_by_fixed = 1` и задан `default_payment_by`, бот пропускает вопрос оплаты.

Последствия:

Плюсы:

* меньше вопросов менеджеру;
* меньше ошибок в заявках;
* правила оплаты задаются в справочнике салона.

Минусы:

* нужно следить, чтобы `default_payment_by` был задан при включённом `is_payment_by_fixed`;
* раннее рабочее название `payment_by_locked` не совпадает с текущим именем в коде, поэтому документация должна использовать фактическое имя.

## ADR-010: Production-деплой сначала простой: VPS + Node.js + SQLite

Статус: принято

Контекст:

MVP нужно быстро и надёжно запустить для реального использования.

Решение:

Первый production-деплой делать на VPS:

* Node.js;
* SQLite-файл;
* systemd или pm2;
* nginx reverse proxy;
* Basic Auth или другой закрытый доступ к админке;
* backup SQLite.

Последствия:

Плюсы:

* понятная эксплуатация;
* дешёвый запуск;
* минимум инфраструктуры;
* быстрое восстановление из backup.

Минусы:

* нужно вручную настроить сервер;
* нужно следить за backup;
* при росте нагрузки может понадобиться более сложная инфраструктура.

## ADR-011: Устойчивый запуск Telegram-бота и best-effort callbacks

Дата: 2026-06-28

Статус: Accepted

Контекст:

В MVP бот работает через Telegraf long polling, а wizard-черновики хранятся in-memory. При временной ошибке Telegram API на `getMe` процесс не должен падать с первой попытки. После рестарта старые inline-кнопки могут ссылаться на потерянный in-memory черновик или иметь невалидный callback query id.

Решение:

Запуск бота выполняется с ограниченным retry. Long polling стартует с `dropPendingUpdates: true`. Callback acknowledgement через `answerCbQuery` выполняется best-effort: ошибки старого или невалидного callback query не роняют процесс, остальные ошибки проходят через общий error handler Telegraf.

Последствия:

Плюсы:

* временный сетевой сбой Telegram API не валит процесс с первой попытки;
* старые inline-кнопки после рестарта не роняют бот;
* in-memory wizard остаётся простым MVP-решением.

Минусы:

* updates, отправленные во время простоя бота, могут быть отброшены;
* незавершённые wizard-черновики после рестарта всё ещё теряются;
* persistent drafts в SQLite остаются отдельной будущей задачей.

## ADR-012: Схлопываем миграции перед первым VPS-деплоем

Дата: 2026-06-28

Статус: Accepted

Контекст:

До первого production-деплоя миграции отражали локальную историю развития схемы: создание начальной схемы, последующие ALTER TABLE, уточнения seed-данных и правила салонов. Production-базы ещё нет, поэтому поддерживать длинную историю локальных промежуточных миграций не обязательно.

Решение:

Заменить активную историю миграций на две чистые миграции:

* `001_init_schema.sql` — вся актуальная схема SQLite;
* `002_seed_initial_data.sql` — начальные справочники MVP.

`orders.final_price` оставить в схеме как зарезервированное поле, но не использовать в бизнес-логике и UI.

Последствия:

Плюсы:

* новая пустая production-база создаётся из понятной актуальной схемы;
* seed-данные отделены от схемы;
* меньше риска протащить локальные промежуточные решения в первый VPS-деплой.

Минусы:

* существующие локальные SQLite-файлы с записями в `schema_migrations` могут требовать пересоздания или ручной синхронизации;
* после первого production-деплоя активные миграции уже нельзя схлопывать без отдельного плана.

## ADR-013: Раздельные Telegram-боты для local development и production

Дата: 2026-06-29

Статус: Accepted

Контекст:

Production-бот ZamerFlow уже работает на VPS через Telegram long polling. Один Telegram bot token нельзя безопасно использовать одновременно в production и локальной разработке: два процесса будут конкурировать за updates, что может приводить к конфликтам, потерянным updates и нестабильным тестам.

Решение:

Использовать разные Telegram-боты и разные SQLite-базы для разных окружений.

* production использует production bot token и production SQLite на VPS;
* local development использует отдельный dev-бот `@zamerflow_dev_bot` и локальную SQLite-базу;
* production bot token не используется локально для тестирования;
* локальное тестирование не требует остановки production-сервиса на VPS;
* `APP_ENV` фиксирует контур приложения;
* `BOT_ENABLED=false` можно использовать для запуска одной админки без Telegram long polling.

Последствия:

Плюсы:

* production не ломается во время локальных тестов;
* можно безопасно тестировать bot flow локально;
* окружения проще различать по `.env` и базе данных;
* локальная проверка админки может идти без Telegram polling.

Минусы:

* нужно поддерживать второго Telegram-бота в BotFather;
* dev-бот может требовать отдельной настройки команд/описания;
* тестовые данные dev-базы могут отличаться от production;
* нужно внимательно проверять `.env` перед запуском.

## ADR-014: Желаемое время замера храним как свободный текст

Дата: 2026-06-29

Статус: Accepted

Контекст:

В реальных заявках менеджеры могут указывать желаемое время прибытия замерщика разными формулировками: `утром`, `первая половина дня`, `с 13 до 15`, `в 19:00`, `после 16`. Олег передаёт это в карточку строго как написано, без интерпретации. Также рядом с датой может встречаться пометка `(нет)`, если менеджер дату не указала или дата была выбрана/согласована отдельно.

Решение:

Хранить желаемое время в `orders.measure_time` как свободный текст. В Telegram wizard добавить optional step после даты. Бот не парсит и не нормализует время. Если время заполнено, первая строка карточки формируется как `На {date} {time} ❗`. Если время не заполнено, формат остаётся прежним: `На {date}`.

Пометка `(нет)` на MVP остаётся частью `measure_date`.

`formatted_card_text` остаётся snapshot итоговой карточки заявки.

Последствия:

Плюсы:

* карточка передаёт формулировку менеджера без искажений;
* не создаём ложную точность;
* не усложняем MVP календарями и интервалами;
* `formatted_card_text` сохраняет итоговый snapshot.

Минусы:

* нельзя надёжно фильтровать заявки по точному времени;
* нельзя строить календарные интервалы без будущей нормализации;
* аналитика по времени потребует отдельного решения позже.

## ADR-015: MVP CI/CD через GitHub Actions и ручной SSH deploy

Дата: 2026-06-29

Статус: Accepted

Контекст:

Production уже работает на VPS. На сервере есть deploy script `/usr/local/sbin/zamerflow-deploy.sh`, который выполняет backup SQLite, обновление кода, установку зависимостей, typecheck, restart сервиса, healthcheck и вывод логов. Для MVP нужен минимальный CI/CD без Docker, Kubernetes, self-hosted runner и сложной release-инфраструктуры.

Решение:

Использовать GitHub Actions:

* `CI` автоматически запускается на `push` в `main` и `pull_request`;
* `Deploy Production` запускается только вручную через `workflow_dispatch`;
* перед deploy выполняются `npm ci` и `npm run typecheck`;
* deploy job подключается к VPS по SSH и вызывает `sudo /usr/local/sbin/zamerflow-deploy.sh`;
* production secrets хранятся в GitHub Secrets и на VPS, но не в репозитории;
* backup SQLite остаётся обязанностью серверного deploy script.

Отклонённые альтернативы:

* self-hosted runner на production VPS;
* автодеплой на каждый push;
* Docker registry;
* Kubernetes deploy;
* blue-green deploy;
* staging/prod matrix.

Последствия:

Плюсы:

* CI защищает `main` базовой проверкой TypeScript;
* production deploy остаётся ручным и контролируемым;
* production logic остаётся на VPS в одном deploy script;
* не нужен self-hosted runner с доступом к production-файлам;
* схема простая и соответствует MVP.

Минусы:

* деплой зависит от доступности VPS по SSH;
* GitHub Secrets нужно настроить вручную;
* server-side deploy script остаётся отдельной эксплуатационной ответственностью;
* нет автоматического staging и advanced rollback.
