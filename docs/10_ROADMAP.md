# 10. Roadmap

## Now

Ближайший фокус — стабилизировать MVP и подготовить production-деплой.

Задачи:

* стабилизировать текущий Telegram bot flow;
* привести документацию в порядок;
* зафиксировать архитектуру;
* проверить актуальность миграций;
* проверить snapshot-логику;
* проверить `salon_required_items`;
* подготовить VPS-деплой;
* добавить защиту админки;
* добавить backup SQLite;
* добавить базовое логирование;
* добавить healthcheck;
* проверить, что `.env`, токены и SQLite-файлы не попадают в git.

## Next

После стабилизации MVP:

* CI/CD через GitHub Actions;
* production service через systemd или pm2;
* nginx reverse proxy;
* Basic Auth для админки;
* Telegram-уведомления владельцу о критических ошибках;
* статистика в админке;
* управление правилами салонов в более удобном UI;
* фильтры по заявкам;
* улучшение страницы заявки;
* ручной smoke-test checklist.

## Later

Когда основной flow будет стабилен:

* AI-парсинг входящих сообщений;
* интеграции с другими мессенджерами;
* расширение CRM-функциональности;
* роли и права в админке;
* импорт/экспорт справочников;
* аналитика по салонам и заявкам;
* PostgreSQL при появлении реальной нагрузки или требований;
* более сложный monitoring stack при необходимости.

## Не делать преждевременно

Не тратить время на:

* микросервисы;
* Kubernetes;
* PostgreSQL без причины;
* ORM;
* React/Vue frontend;
* сложную дизайн-систему;
* сложный CI/CD;
* AI до стабилизации базового flow.

## Критерий готовности MVP

MVP можно считать рабочим, когда:

* менеджер может авторизоваться;
* менеджер может создать заявку;
* карточка заявки корректна;
* заявка сохраняется в SQLite;
* Олег может посмотреть заявку в админке;
* салонные правила применяются автоматически;
* админка защищена на production;
* есть backup SQLite;
* есть понятная инструкция деплоя;
* есть базовая проверка работоспособности.
