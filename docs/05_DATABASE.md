# 05. Database

## Технология

ZamerFlow использует SQLite через `better-sqlite3`.

База хранится в файле по пути из переменной окружения:

```text
DATABASE_PATH
```

Значение по умолчанию:

```text
./data/zamerflow.sqlite
```

При создании подключения приложение включает внешние ключи:

```sql
PRAGMA foreign_keys = ON;
```

## Миграции

Миграции лежат в папке:

```text
migrations/
```

При старте приложение:

1. создаёт таблицу `schema_migrations`, если её нет;
2. читает `.sql`-файлы из `migrations`;
3. сортирует их по имени;
4. применяет только новые миграции;
5. записывает применённый filename в `schema_migrations`.

Если миграция сама открывает `BEGIN TRANSACTION`, runner добавляет запись в `schema_migrations` перед финальным `COMMIT`.

Перед первым VPS-деплоем история локальных миграций была схлопнута в две чистые миграции:

* `001_init_schema.sql` — актуальная схема БД, таблицы, внешние ключи и индексы;
* `002_seed_initial_data.sql` — начальные справочные данные для MVP.

Это допустимо только до первого production-деплоя. После появления production-базы старые применённые миграции нельзя переписывать или удалять из активной истории без отдельного плана миграции данных.

## Основные таблицы

### `salons`

Справочник салонов.

Хранит:

* название;
* alias;
* email;
* базовую цену;
* диапазон доппозиций;
* стоимость километража;
* тип оплаты по умолчанию;
* признак фиксированной оплаты;
* текст условий оплаты;
* комментарий к цене;
* сортировку;
* активность.

Фактическое поле фиксированной оплаты в текущем коде:

```text
is_payment_by_fixed
```

Раннее рабочее название было `payment_by_locked`, но в текущей схеме и коде используется `is_payment_by_fixed`.

### `salon_managers`

Менеджеры салонов.

Хранит:

* `salon_id`;
* имя;
* телефон;
* email;
* должность;
* роль менеджера;
* сортировку;
* активность.

Роль менеджера сейчас используется как `manager` или `director`.

### `manager_auth_codes`

Одноразовые коды доступа.

Хранит:

* `auth_code`;
* `manager_id`;
* дату создания;
* дату использования;
* `is_used`.

Код используется для первичной привязки Telegram-пользователя к менеджеру.

### `telegram_users`

Связь Telegram-пользователя с системой.

Хранит:

* Telegram user id;
* username;
* first_name;
* last_name;
* role;
* `salon_id`;
* `manager_id`;
* активность.

После авторизации менеджера бот использует эту таблицу, чтобы не спрашивать салон и менеджера при создании заявки.

### `orders`

Заявки на замер.

Хранит:

* статус;
* ссылки на салон и менеджера;
* snapshot-поля салона и менеджера;
* дату замера;
* желаемое время прибытия замерщика;
* адрес;
* метро;
* контакт клиента;
* тип оплаты;
* тарифные snapshot-поля;
* `final_price`;
* доплаты;
* комментарий;
* готовый текст карточки;
* Telegram user id.

`final_price` сейчас зарезервирован под будущее, не используется в бизнес-логике и должен сохраняться `NULL`.

Поле остаётся в схеме для совместимости и будущего развития, но не показывается в карточке и админке как итоговая стоимость.

`orders.measure_time` — свободный текст пожелания по времени прибытия замерщика.

Он не нормализуется и не парсится. Может содержать `утром`, `первая половина дня`, `с 13 до 15`, `в 19:00`, `после 16` и другие формулировки. Значение передаётся в карточку строго как написано.

### `order_items`

Позиции заявки.

Хранит:

* `order_id`;
* тип позиции;
* количество;
* комментарий;
* snapshot-название;
* snapshot-цену;
* snapshot-текст цены;
* snapshot-строку карточки;
* признак автоматического добавления;
* source;
* сортировку.

Для пользовательских позиций source обычно `user_input`.

Для обязательных салонных позиций source должен быть `salon_required_item`.

### `salon_required_items`

Правила обязательных позиций салона.

Хранит:

* `salon_id`;
* `item_type`;
* `item_name`;
* `unit_price`;
* `price_text`;
* `card_text`;
* `quantity`;
* `is_required`;
* `auto_add_to_order`;
* `comment`;
* `sort_order`;
* `is_active`.

Активные записи с `auto_add_to_order = 1` автоматически попадают в заявку.

## Snapshot-логика

Snapshot-поля нужны, чтобы старая заявка не менялась при изменении справочника.

Примеры:

* `salon_name_snapshot`;
* `salon_email_snapshot`;
* `manager_name_snapshot`;
* `manager_phone_snapshot`;
* `manager_role_snapshot`;
* `item_name_snapshot`;
* `unit_price_snapshot`;
* `price_text_snapshot`;
* `card_text_snapshot`.

Если у салона поменялся email или тариф, старые заявки должны показывать старые данные.

## Итоговая цена

`orders.final_price` пока не использовать.

Правила:

* не рассчитывать;
* не заполнять;
* не показывать как итог;
* не строить бизнес-логику вокруг него.

Сейчас итоговую стоимость считает замерщик на объекте.

## Фиксированная оплата

Текущее фактическое поле:

```text
is_payment_by_fixed
```

Вместе с:

```text
default_payment_by
```

Если `is_payment_by_fixed = 1` и задан `default_payment_by`, бот не спрашивает, кто оплачивает замер.

## Правила миграций

* Новые изменения схемы делать только через новые SQL-файлы.
* Не редактировать старые применённые миграции, если база уже могла быть создана у пользователя.
* Для SQLite осторожно использовать `ALTER TABLE`.
* Не удалять данные без явной причины.
* Для справочников использовать `is_active`, а не физическое удаление, если это бизнес-сущность.
* Для сортируемых справочников использовать `sort_order`.

## Что нельзя коммитить

Нельзя коммитить:

* `.env`;
* SQLite-файлы;
* journal/wal/shm-файлы SQLite;
* токены;
* production backup-файлы;
* персональные dump-файлы.

Сейчас `.gitignore` уже закрывает `.env`, `data/*.sqlite`, `data/*.sqlite-journal`, `node_modules` и `dist`.

Рекомендуется дополнить позже:

```gitignore
data/*.sqlite-*
logs/
backups/
```
