# 09. Security

## Основные риски

ZamerFlow хранит и обрабатывает персональные и рабочие данные:

* адреса клиентов;
* телефоны клиентов;
* телефоны менеджеров;
* email салонов;
* тексты заявок;
* Telegram user id;
* коды доступа;
* токен Telegram-бота.

## Персональные данные и роли

В ZamerFlow персональными данными считаются:

* данные клиента: имя или контактное лицо, телефон, адрес замера, комментарий к заявке;
* данные менеджера: имя, телефон, Telegram user id / username, привязка к салону;
* данные салона: название, email, настройки правил, если они позволяют связаться с конкретным человеком.

ZamerFlow не должен собирать специальные категории персональных данных, биометрию, паспортные данные, сведения о здоровье, политических взглядах, религии и другие данные, не нужные для оформления заявки на замер.

Перед production должен быть определён оператор персональных данных.

Для MVP предполагается, что оператором является бизнес-владелец процесса оформления заявок. Разработчик или администратор сервера выполняет техническую обработку данных по поручению оператора.

## Основание обработки персональных данных

ZamerFlow не должен использовать персональные данные для целей, не связанных с оформлением и сопровождением заявки на замер.

Перед production оператор должен определить правовое основание обработки персональных данных клиентов и менеджеров:

* согласие;
* договор;
* заявка клиента;
* иное основание, выбранное оператором.

ZamerFlow не должен рассылать маркетинговые сообщения клиентам и не должен передавать персональные данные третьим лицам, кроме случаев, необходимых для выполнения заявки или отдельно согласованных оператором.

## `.env`

`BOT_TOKEN` и другие секреты должны храниться только в `.env` или production secret storage.

Файл `.env` нельзя коммитить.

В репозитории допустим только `.env.example`.

## Telegram bot token

`BOT_TOKEN` даёт доступ к управлению ботом.

Правила:

* production `BOT_TOKEN` нельзя использовать локально, пока production-сервис запущен на VPS;
* local development использует отдельный токен dev-бота `@zamerflow_dev_bot`;
* dev `BOT_TOKEN` и production `BOT_TOKEN` должны храниться только в соответствующих `.env`;
* не писать токен в README;
* не вставлять токен в issue/PR/chat;
* не логировать токен;
* при утечке сразу перевыпустить токен через BotFather.

Один Telegram bot token нельзя безопасно использовать одновременно в production и local development: два long polling процесса будут конкурировать за updates.

## Dispatch-чат

Dispatch-чат содержит персональные данные из карточек заявок: адреса, телефоны клиентов, комментарии и рабочие детали замера. Служебный dispatch-заголовок также может содержать исходный адрес менеджера и нормализованный адрес DaData, поэтому чат должен оставаться закрытым рабочим чатом.

Правила:

* бот должен быть добавлен только в закрытый рабочий Telegram-чат;
* `DISPATCH_CHAT_ID` должен храниться в `.env` нужного окружения, а не в коде;
* `DISPATCH_CHAT_ID` не является токеном, но это конфигурация окружения и её не нужно hardcode-ить;
* `DISPATCH_CHAT_ID` должен указывать на группу или супергруппу, а не на личный чат пользователя с ботом;
* не коммитить реальные значения `DISPATCH_CHAT_ID`;
* не логировать полный текст карточки;
* не логировать адреса и телефоны клиентов;
* не добавлять в dispatch-чат лишних участников.

## DaData keys

`DADATA_API_KEY` и `DADATA_SECRET_KEY` являются секретами.

Правила:

* хранить ключи только в `.env`, production secret storage или GitHub Secrets, если они когда-нибудь понадобятся CI/CD;
* не коммитить ключи;
* не вставлять ключи в README, docs, issue, PR или chat;
* не логировать ключи;
* если ключи попали в чат, PR или logs, перевыпустить их в DaData.

Адреса клиентов являются персональными данными. При ошибках DaData нельзя логировать полный адрес, request body или нормализованный адрес. Для диагностики можно логировать статус ошибки и длину адреса.

## SQLite-файлы

SQLite database files нельзя коммитить.

Нельзя коммитить:

* `data/*.sqlite`;
* `data/*.sqlite-journal`;
* `data/*.sqlite-wal`;
* `data/*.sqlite-shm`;
* `data/*.sqlite-*`;
* `data/*.db`;
* `data/*.db-*`;
* `logs/`;
* `backups/`;
* backup-файлы с production-данными.

Production SQLite нельзя копировать в локальную разработку без осознанного sanitized backup. Локальная разработка должна использовать отдельную dev-базу, например `./data/zamerflow-dev.sqlite`.

В `.gitignore` должны быть закрыты:

```gitignore
data/*.sqlite
data/*.sqlite-*
data/*.db
data/*.db-*
backups/
logs/
```

## Локализация production-данных

Если ZamerFlow используется для обработки персональных данных граждан РФ, production SQLite database и production backups должны размещаться на инфраструктуре в РФ, если иное решение не принято отдельно с юридической оценкой.

Запрещено размещать production SQLite database и backups с персональными данными граждан РФ на зарубежных VPS без отдельного решения по трансграничной передаче персональных данных.

## Админка

Админку нельзя открывать в интернет без защиты.

Пока авторизация в приложении не реализована, production-доступ должен быть закрыт внешним способом:

* nginx Basic Auth;
* VPN;
* SSH tunnel;
* firewall allowlist.

Минимальный вариант MVP — Basic Auth на nginx и закрытый прямой порт Fastify.

## Коды доступа

`manager_auth_codes` — одноразовые коды авторизации менеджеров.

Правила:

* коды должны быть достаточно непредсказуемыми;
* использованный код не должен подходить повторно;
* сброс кода через админку должен использоваться только осознанно;
* коды не стоит логировать.

## Логи

Не логировать:

* полный текст карточек;
* адреса клиентов;
* телефоны клиентов;
* auth codes;
* токены;
* `.env`;
* SQL dumps.

Для диагностики использовать ids:

* `order_id`;
* `salon_id`;
* `manager_id`;
* `telegram_user_id`.

## Backup

Backup SQLite содержит персональные данные.

Правила:

* хранить backup в закрытой директории;
* не отдавать backup через nginx;
* ограничить права файлов;
* ротировать старые backup;
* не переносить backup в публичные каналы.

## Сроки хранения и удаление персональных данных

Персональные данные должны храниться не дольше, чем требуется для оформления заявки, сопровождения замера, истории взаимодействия с салоном и разрешения спорных ситуаций.

Для MVP допускается хранить заявки без автоматического удаления, но перед production нужно определить:

* срок хранения заявок;
* срок хранения backups;
* кто может инициировать удаление;
* как обрабатывается запрос клиента на удаление или уточнение данных.

Snapshot-поля старых заявок нужны для сохранения истории заявки на момент создания. Будущая реализация архивации или удаления должна учитывать snapshot-логику и удалять или архивировать персональные данные осознанно, без поломки карточек и истории спорных ситуаций.

## VPS-доступ

Production VPS должен использовать:

* SSH key;
* отключение password login, если возможно;
* отдельного пользователя `zamerflow`;
* минимальные права;
* закрытые лишние порты;
* регулярные обновления системы.

## GitHub

Не хранить в GitHub:

* `.env`;
* production database;
* backup;
* токены;
* приватные ключи;
* персональные выгрузки.

Для CI/CD использовать GitHub Secrets.

## CI/CD

Production secrets хранить только в GitHub Secrets и на VPS.

Для GitHub Actions deploy нужны:

* `VPS_HOST`;
* `VPS_PORT`;
* `VPS_USER`;
* `VPS_SSH_KEY`;
* `VPS_KNOWN_HOSTS`.

Правила:

* приватный SSH-ключ нельзя коммитить;
* production `.env` нельзя передавать в workflow;
* deploy-пользователь на VPS должен иметь минимальные права;
* желательно разрешить deploy-пользователю через `sudo` только команду `/usr/local/sbin/zamerflow-deploy.sh`;
* в logs workflow нельзя выводить `.env`, токены, SQLite backup, SQL dumps и приватные ключи;
* GitHub Actions должен только вызывать серверный deploy script, а не хранить всю production-логику в YAML.

## Уведомление Роскомнадзора

Перед production-использованием на реальных заявках оператор персональных данных должен оценить необходимость подачи уведомления в Роскомнадзор.

Это организационная обязанность оператора, а не автоматическая функция ZamerFlow.

В production checklist должно быть понятно:

* кто оператор персональных данных;
* подано ли уведомление или почему применяется исключение;
* кто отвечает за актуальность сведений.

## Инциденты и утечки персональных данных

Инцидентами считаются:

* утечка SQLite database или backup;
* публикация `.env`, токенов или приватных ключей;
* открытая админка с доступом к заявкам;
* попадание адресов, телефонов или карточек замера в публичные логи, issue, PR, чаты или внешние сервисы;
* несанкционированный доступ к VPS или GitHub Secrets.

При подозрении на инцидент:

1. ограничить доступ к скомпрометированному ресурсу;
2. сохранить технические следы для расследования;
3. перевыпустить скомпрометированные токены/ключи;
4. оценить состав затронутых персональных данных;
5. уведомить оператора персональных данных;
6. оператор должен оценить обязанность уведомления Роскомнадзора в установленные законом сроки.

## MVP-минимум перед production

Перед открытием production нужно:

* определить оператора персональных данных;
* закрыть админку Basic Auth/VPN/SSH tunnel;
* проверить `.gitignore`;
* убедиться, что `.env` не попал в git history;
* убедиться, что SQLite-файлы не попали в git;
* убедиться, что production SQLite и backups размещены на инфраструктуре в РФ, если обрабатываются персональные данные граждан РФ;
* настроить backup;
* определить срок хранения заявок и backups;
* определить порядок обработки запроса на удаление/уточнение персональных данных;
* оценить необходимость уведомления Роскомнадзора;
* зафиксировать, кто имеет доступ к админке и production VPS;
* проверить, что production-логи не содержат адресов, телефонов и полных карточек заявок;
* ограничить доступ к VPS.
