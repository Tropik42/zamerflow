# 03. Telegram Bot

## Назначение

Telegram-бот нужен менеджерам салонов для оформления заявок на замеры.

Бот должен:

* авторизовать менеджера;
* понять его салон и manager_id;
* собрать заявку;
* применить правила салона;
* сформировать карточку;
* сохранить заявку.

## Запуск

Бот запускается вместе с приложением через:

```bash
npm run dev
```

Используется Telegraf long polling.

Webhook в MVP не используется.

При запуске long polling бот может отбрасывать pending updates после простоя процесса. Это MVP-компромисс: сообщения и кнопки, отправленные во время простоя, могут быть потеряны, но старые callback updates после рестарта не должны ронять процесс.

## Команды

### `/start`

В личном чате:

* проверяет, привязан ли Telegram-пользователь к менеджеру;
* если пользователь уже авторизован, показывает его менеджера и салон;
* если нет — просит ввести одноразовый код доступа.

В группе:

* сообщает, что для оформления заявки нужно использовать `/new` и отвечать reply-сообщениями.

### `/new`

Запускает создание новой заявки.

Если пользователь не авторизован, бот просит сначала пройти `/start`.

Если пользователь авторизован как менеджер или директор салона, бот:

* берёт салон из `telegram_users`;
* берёт менеджера из `telegram_users`;
* не спрашивает выбор салона;
* создаёт черновик сразу с salon/manager snapshot.

Для других ролей или будущего оператора может быть доступен выбор активного салона из списка.

### `/whoami`

Показывает текущую привязку Telegram-пользователя:

* telegram_user_id;
* username;
* роль;
* роль в салоне;
* салон;
* менеджер.

### `/cancel`

Отменяет текущую заявку и удаляет in-memory черновик.

## Авторизация менеджера

Менеджер авторизуется через одноразовый код доступа.

Код хранится в `manager_auth_codes` и связан с `manager_id`.

Flow:

```text
/start
→ бот проверяет telegram_users
→ если привязки нет, просит код
→ менеджер вводит код
→ бот ищет неиспользованный код
→ проверяет активного менеджера и активный салон
→ создаёт telegram_users
→ помечает код использованным
```

Код после успешного использования становится `is_used = 1`.

## Черновики

Черновики заявок хранятся in-memory.

Ключ черновика строится по Telegram chat id и user id.

Следствие: при рестарте процесса незавершённые черновики теряются. Для MVP это допустимо.

Старые inline-кнопки после рестарта считаются best-effort: если Telegram уже не принимает callback acknowledgement, бот не должен падать. Пользователь может получить сообщение "Черновик не найден. Начните новую заявку через /new." или начать новую заявку вручную.

## Основной flow заявки

Для авторизованного менеджера бот спрашивает:

1. Контакт клиента или представителя.
2. Точный адрес клиента с населённым пунктом.
3. Метро / ориентир.
4. Дату замера.
5. Позиции под замер.
6. Кто оплачивает, если оплата не фиксирована салоном.
7. Доплаты / особенности.
8. Комментарий.
9. Предпросмотр карточки.
10. Принятие заявки.

В вопросе оплаты кнопки показываются в именительном падеже: "клиент", "салон", "депозит", "с ИП на самозанятого". В карточке после "Оплата" используется форма для карточки из справочника, например "клиентом", "салоном", "депозитом".

## Позиции под замер

Сейчас доступны позиции:

* кухня;
* шкаф;
* ниша под шкаф;
* тумба под раковину;
* гардеробная;
* инсталляция;
* ТВ-зона;
* рабочая зона;
* стеновые панели;
* по плану;
* вписать вручную.

Пользователь может выбрать несколько позиций.

Кнопки и форма стандартной позиции для карточки берутся из общего справочника `src/bot/measureServiceItems.ts`. В кнопках используется именительный падеж, а после слова "Замер" в карточке используется форма для карточки, например "кухни", "шкафа", "гардеробной". Вариант "вписать вручную" просит менеджера ввести произвольный текст позиции; этот текст сохраняется в заявке как введённая позиция.

## Обязательные позиции салона

Перед вопросами бот загружает активные `salon_required_items` для салона.

Если позиция активна и `auto_add_to_order = 1`, она автоматически добавляется в черновик.

Для таких позиций сохраняются snapshot-поля:

* `itemNameSnapshot`;
* `unitPriceSnapshot`;
* `priceTextSnapshot`;
* `cardTextSnapshot`;
* `isAutoAdded`;
* `source = salon_required_item`.

## Фиксированная оплата

В текущем коде используется:

* `default_payment_by`;
* `is_payment_by_fixed`.

Если `is_payment_by_fixed = 1` и задан `default_payment_by`, бот не спрашивает оплату, а сразу использует значение из салона.

Пример: `Settee` и `Тетрис Кухни` могут иметь фиксированную оплату салоном.

## Предпросмотр карточки

Перед сохранением бот формирует карточку и показывает её пользователю.

Действия:

* принять;
* заполнить заново;
* отменить.

После принятия заявка сохраняется в SQLite со статусом `accepted`.

## Работа в группе

В группе бот работает через команды и reply-сообщения.

Для текстовых шагов пользователь должен отвечать reply-сообщением на вопрос бота. Это нужно, чтобы flow работал даже при включённом Telegram privacy mode.

Кнопочные шаги работают через inline-кнопки.

## Что бот пока не делает

На MVP бот не делает:

* AI-парсинг произвольных сообщений;
* автоматическую отправку карточки в другой чат;
* webhook;
* интеграции с другими мессенджерами;
* расчёт итоговой стоимости.
