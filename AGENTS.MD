# AGENTS.md

## Проект

ZamerFlow — это Telegram-бот и server-rendered админка для оформления заявок на замеры кухонь, шкафов и встроенной мебели.

Проект делается как MVP под реальный процесс Олега: салоны и менеджеры создают структурированные заявки, система формирует карточку замера, а админка позволяет управлять салонами, менеджерами, кодами доступа, заявками и правилами салонов.

## Язык

Пользовательские тексты, документацию, README, тексты бота и админки писать на русском языке.

Идентификаторы в коде, названия файлов, функций, переменных и технические сущности можно писать на английском.

## Принципы MVP

Сначала нужен надёжный работающий продукт для Олега и тестовых менеджеров.

Проект должен оставаться простым:

* один репозиторий;
* один Node.js-процесс на MVP;
* Telegram-бот;
* Fastify-админка;
* одна SQLite-база;
* SQL-миграции.

Бизнес-логика должна жить в services/repositories, а не размазываться по Telegram handlers и Fastify routes.

Желаемый слой:

Telegram bot / Admin UI
→ services
→ repositories
→ SQLite

## Основные бизнес-сущности

Основные таблицы и сущности:

* `salons`
* `salon_managers`
* `manager_auth_codes`
* `telegram_users`
* `orders`
* `order_items`
* `salon_required_items`

Менеджер авторизуется в Telegram по одноразовому коду доступа.

После авторизации бот знает:

* `telegram_user_id`
* `salon_id`
* `manager_id`

При создании заявки менеджером:

* салон не спрашивать;
* менеджера не спрашивать;
* брать `salon_id` и `manager_id` из `telegram_users`;
* спрашивать только переменные поля заявки.

## Snapshot-логика

В заявках нужны snapshot-поля, чтобы старая заявка не менялась при изменении справочников.

Не удалять snapshot-логику.

Примеры snapshot-полей:

* `salon_name_snapshot`
* `salon_email_snapshot`
* `manager_name_snapshot`
* `manager_phone_snapshot`
* ценовые snapshot-поля
* `card_text_snapshot` для обязательных позиций

Если у салона или менеджера позже поменяется email, телефон, название или тарифы, старые заявки должны отображаться так, как они были созданы изначально.

## Логика оплаты

У некоторых салонов оплата фиксирована.

Для этого используются поля:

* `default_payment_by`
* `payment_by_locked`

Если `payment_by_locked = 1`, бот не должен спрашивать, кто оплачивает замер. Он должен использовать `default_payment_by`.

Примеры:

* `Settee` — оплата салоном.
* `Тетрис Кухни` — оплата салоном.

## Итоговая цена

`orders.final_price` пока оставить в схеме, но не использовать.

Не рассчитывать итоговую стоимость заявки.

Не показывать “итого” в карточке заявки.

Причина: итоговую стоимость считает замерщик на объекте, потому что на месте могут появиться дополнительные позиции или уточнения.

## Обязательные позиции салонов

У некоторых салонов есть обязательные дополнительные позиции, которые должны автоматически попадать в заявку.

Для этого используется таблица:

* `salon_required_items`

Пример:

Салон `Тетрис Кухни` всегда требует:

`Лифтовой холл + подъезд + паркинг`

Цена: `1,000₽`.

Смысл: нужно проверить, пройдёт ли крупногабаритная мебель через лифтовой холл, подъезд и паркинг.

Это не свободный комментарий, а правило салона.

Обязательные позиции салона должны автоматически добавляться в `order_items` при создании заявки.

Для автоматически добавленных позиций сохранять snapshot-поля:

* `item_name_snapshot`
* `unit_price_snapshot`
* `price_text_snapshot`
* `card_text_snapshot`
* `is_auto_added`
* `source`

`source` для таких позиций: `salon_required_item`.

## Формат карточки заявки

Карточка заявки обычно состоит из трёх блоков.

Первый блок:

* дата и время;
* название салона;
* email салона;
* менеджер или директор;
* телефон менеджера.

Второй блок:

* адрес;
* метро или ориентир;
* контакт клиента, прораба или дизайнера;
* что замерять;
* комментарии и особенности.

Третий блок:

* кто оплачивает;
* стартовая стоимость;
* доппозиции;
* километраж;
* специальные условия салона.

Карточка не должна показывать итоговую стоимость, пока это явно не будет добавлено отдельным решением.

## Админка

Админка работает на Fastify и server-rendered HTML.

Не делать SPA.

Не добавлять React/Vue.

Минимальные разделы админки:

* `/admin`
* `/admin/salons`
* `/admin/managers`
* `/admin/auth-codes`
* `/admin/orders`
* управление `salon_required_items`

Управление обязательными позициями можно сделать:

* либо отдельной страницей;
* либо блоком на странице редактирования салона.

Локально админка может работать без авторизации.

Перед production-деплоем админка должна быть защищена.

Минимальная защита для MVP:

* Basic Auth;
* или доступ только через VPN/SSH tunnel;
* или другой простой закрытый доступ.

## База данных

Использовать SQLite.

Использовать `better-sqlite3`.

Использовать SQL-миграции.

Использовать prepared statements.

Не удалять и не перезаписывать существующие данные в миграциях без явной просьбы.

Осторожно использовать `ALTER TABLE`, учитывая ограничения SQLite.

Сохранять стиль текущей схемы:

* `create_datetime`
* `modify_datetime`
* `is_active`
* `sort_order`, если сущность сортируется в админке или карточке.

Не коммитить локальные SQLite-файлы.

В `.gitignore` должны быть:

* `.env`
* `data/*.sqlite`
* `data/*.sqlite-*`
* `logs/`
* `node_modules/`
* `dist/`

## Деплой

Цель деплоя: бот и админка должны работать без ноутбука пользователя.

Предпочтительный MVP-деплой:

* VPS
* Node.js
* SQLite-файл на сервере
* systemd или pm2
* nginx reverse proxy для админки
* Basic Auth для админки
* backup SQLite перед деплоем
* регулярный backup SQLite

Не усложнять деплой без необходимости.

Docker можно добавить только если он реально упрощает поддержку. Не добавлять Docker “ради правильности”.

## Логирование и мониторинг

Для MVP достаточно лёгкой observability-схемы:

* структурные логи;
* лог ошибок бота;
* лог запуска и остановки;
* healthcheck endpoint;
* restart on failure через systemd/pm2;
* уведомление владельцу в Telegram при критической ошибке;
* регулярный backup SQLite;
* проверка свободного места на сервере.

Не добавлять Prometheus/Grafana/Sentry без явной просьбы.

## CI/CD

Предпочтительно использовать GitHub Actions.

Базовый pipeline:

* `npm ci`
* `npm run typecheck`
* optional build, если появится build step
* deploy по SSH
* backup SQLite перед деплоем
* применение миграций
* restart сервиса
* проверка health endpoint

Секреты не хранить в репозитории.

Использовать GitHub Secrets для:

* SSH key;
* host;
* username;
* production env values, если понадобится.

## Проверки перед завершением задачи

Перед тем как считать задачу завершённой:

* запустить `npm run typecheck`, если возможно;
* если проверка не запускалась — явно написать об этом;
* проверить, что не сломан основной Telegram flow;
* проверить, что не сломана админка;
* проверить, что миграции идут в правильном порядке;
* проверить, что `.env`, токены и SQLite-файлы не попали в git;
* обновить документацию, если изменилось поведение системы.
