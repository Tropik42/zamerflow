# ZamerFlow Bot

Backend-проект Telegram-бота для оформления заявок на замеры кухонь и встроенной мебели.

## Установка

```bash
npm install
```

## Настройка окружения

Создайте `.env` из примера:

```bash
cp .env.example .env
```

Вставьте токен Telegram-бота в переменную `BOT_TOKEN`:

```env
BOT_TOKEN=ваш_токен_бота
DATABASE_PATH=./data/zamerflow.sqlite
ADMIN_PORT=3000
```

## Запуск

```bash
npm run dev
```

Бот запускается через long polling. Webhook не используется.

Вместе с ботом запускается локальная HTML-админка:

```text
http://localhost:3000/admin
```

Если нужен другой порт, задайте `ADMIN_PORT` в `.env`. Авторизация админки пока не реализована: используйте её только для локального тестирования и не открывайте наружу.

Пошаговое заполнение заявки работает в личном чате и в группе.

В группе отправьте `/new` и отвечайте на текстовые вопросы reply-сообщениями на вопросы бота. Это позволяет работать даже при включенном Telegram privacy mode. Для выбора позиций, оплаты, плана и финального действия используются inline-кнопки. Для отмены в группе можно отправить `/cancel` или нажать inline-кнопку «❌ Отменить».

## npm scripts

- `npm run dev` — запуск Telegram-бота.
- Админка запускается той же командой `npm run dev`.
- `npm run typecheck` — проверка TypeScript без сборки.

## База данных

Данные сохраняются в SQLite-файл по пути из `DATABASE_PATH`. По умолчанию это:

```text
./data/zamerflow.sqlite
```

Проект сейчас в dev-стадии. Если нужно сбросить базу и применить схему заново, можно удалить файл:

```bash
rm -f data/zamerflow.sqlite data/zamerflow.sqlite-journal
```

При старте приложение включает `PRAGMA foreign_keys = ON`, создает таблицу `schema_migrations` и применяет новые SQL-миграции из папки `migrations`. Новые миграции повторно не применяются, если их имя уже есть в `schema_migrations`.

Схема подготовлена для справочника салонов, менеджеров и заявок: есть таблицы `salons`, `salon_managers`, `telegram_users`, `orders`, `order_items` и `salon_required_items`. Позиции заявки хранятся в `order_items`, а в `orders` есть snapshot-поля для текста салона и менеджера на момент оформления заявки.

`salon_required_items` хранит правила салона, которые автоматически добавляют позиции в заявку. Например, для салона «Тетрис Кухни» в заявку автоматически попадает «Лифтовой холл + подъезд + паркинг» со строкой тарифа `Лифтовой холл + подъезд + паркинг в 1,000₽.`.

`orders.final_price` пока не используется и не рассчитывается. Карточка передаётся замерщику, а итоговую стоимость он считает сам на объекте по тарифам и фактическим доппозициям.

### Тестовый код доступа менеджера

Код доступа заранее связывается с `manager_id`. После успешной авторизации код становится использованным и повторно не подходит.

```sql
-- 1. Найти manager_id
SELECT manager_id, manager_name FROM salon_managers;

-- 2. Создать код
INSERT INTO manager_auth_codes (
  auth_code,
  manager_id,
  create_datetime,
  is_used
)
VALUES (
  'TEST-001',
  1,
  datetime('now'),
  0
);
```

## Что реализовано

- `/start` с авторизацией менеджера через одноразовый код доступа.
- `/new` и кнопка «➕ Новая заявка».
- `/whoami` и кнопка «👤 Кто я».
- `/cancel` для отмены текущей заявки.
- Локальная админка `/admin` для салонов, менеджеров, кодов доступа и просмотра заявок.
- Салонные обязательные позиции, которые автоматически добавляются в заявку и карточку.
- Пошаговое заполнение заявки: сначала выбор активного салона из БД, затем данные замера.
- In-memory черновики по chat id и Telegram user id.
- Выбор нескольких позиций под замер с количеством.
- Предпросмотр карточки заявки.
- Сохранение принятой заявки в SQLite через чистый SQL и `better-sqlite3`.

Данные сохраняются в БД, а проект разделен на слой бота, админки, репозитории и миграции.

В проект не добавлены ORM, Docker, React/Vue, AI/LLM, WhatsApp/MAX, webhook и автоматическая отправка карточки в другой чат.
